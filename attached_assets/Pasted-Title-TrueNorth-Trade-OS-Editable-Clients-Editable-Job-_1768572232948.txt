Client edits

Job sheet edits

Server PATCH route to update allowed fields

Enforce status-based locking rules to protect signed-off jobs

Block or limit edits after sign-off (see assumptions)

Ensure existing sign-off validation rules remain unchanged

clients

updated_at, updated_by (add if missing)

updated_at, updated_by (add if missing)

Auth: admin + works_manager; optionally engineer for own jobs (assumption below)

GET /api/jobs/:id (if needed by editor view)

Clients

ClientDetail view: add “Edit” button

ClientEdit view or modal:

Prefill current values

Save -> optimistic UI via TanStack Query, then refetch

Show success toast + error messages

Jobs

JobDetail/JobSheet view: add “Edit Job Sheet” button

JobEdit view or section:

Editable: description/notes, dates, client reference fields, non-signoff metadata

NOT editable: signatures, evidence photos list integrity, geo sign-off data (unless your existing UI supports adding evidence photos separately)

Rebuilding client portal, quotes/invoices flows, or job lifecycle redesign

New role system (use existing roles)

Full historical diff viewer (unless an audit_events table already exists)

Changing sign-off rules/validation

Assumptions:

Roles: Only admin and works_manager can edit clients; jobs editable by admin/works_manager; engineers can edit ONLY their own jobs while status != Signed Off.

Signed Off jobs are locked: no edits to job core fields after Signed Off. (If you want revisions, agent must Stop & Ask.)

“Job sheet” refers to the jobs table record + related display fields, not quotes/invoices.

Acceptance criteria (testable):

Client: An authorized user can edit a client, save, and see updated values on refresh; unauthorized users receive 403 and UI shows “not permitted”.

Job: An authorized user can edit a job (status not Signed Off), save, and see updated values; unauthorized users receive 403.

Signed Off job: PATCH /api/jobs/:id returns 409 or 403 when status is Signed Off; UI disables editing and explains why.

Existing sign-off completion rules remain intact (evidence photo + both signatures + geolocation are still required to sign off; editing does not bypass or remove them).

updated_at and updated_by are correctly set on both clients and jobs updates.

Stop & Ask:

If the codebase uses different route naming, auth middleware, or schema fields than assumed, pause and ask before implementing.

If product requirements want post–Signed Off edits or revisions, pause and ask for the exact workflow (rev numbers, audit log expectations, who can unlock).

If clients have “contacts” and “properties” tables and you want those editable too, pause and ask which sub-entities are in scope.

Integrations (only if requested):

None. Do not touch OpenAI/Maps/Stripe/Outlook.

Notes:

Validate inputs server-side (Zod if already used with drizzle-zod); never trust client.

Prevent mass assignment: whitelist fields allowed to update.

Ensure TanStack Query cache invalidation is correct for list + detail queries.

Add minimal unit tests if an existing test setup exists; otherwise keep changes minimal.

Prompt cost savers

Implement edit UI as a single shared “ClientForm” and “JobForm” used by create/edit if create already exists.

Avoid new tables unless audit trail is mandatory—use updated_at/updated_by first.
