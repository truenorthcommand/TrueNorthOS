# P0 (Critical) — Stop fetching ALL records every request
## Implement an intent-based context builder
Create a backend module (new files are OK) that builds context based on intent:
- intents: `chat | insights | suggestions | search`

Replace “fetch everything” calls with **bounded, relevant subsets**:
- Jobs: active OR last 25 updated
- Clients: last 25 updated
- Quotes/Invoices: last 50 updated OR only relevant statuses (pending/overdue/etc.)
Prefer aggregate SQL for insights (COUNT/SUM/GROUP BY) instead of pulling full rows.

### Requirements
- All queries must be tenant-scoped (orgId from auth/session, not client-supplied).
- Add hard caps to prevent runaway payloads.
- Ensure response quality remains good (context is smaller but more relevant).

# P0 (Critical) — Token budgeting + truncation
Add a simple budget system (tokenizer optional; char-based acceptable):
- Cap business context string size (e.g. 8–16k chars)
- Cap the number of entities included
- Dynamically reduce conversation history based on remaining budget
- Cap web search injection (top 3 results max, summarized before injection)
- If context would exceed budget, include a short note in logs (sizes/counts only).

This MUST prevent “context length exceeded” style failures.

# P0 (Critical) — Caching for derived context
Add caching for derived summaries/results:
- Org/business summary: TTL ~24h
- Smart insights: TTL 1–6h
- Suggestions: TTL 10–30m

Use Postgres (Drizzle) as cache storage if Redis isn’t present:
- Create an `ai_cache` table: key, value(json/text), orgId, updatedAt, expiresAt, version
- Provide helper functions: `cacheGet(key)`, `cacheSet(key, value, ttl)`, `cacheDelPrefix(prefix)`
- Log cache hit/miss (no sensitive content).
